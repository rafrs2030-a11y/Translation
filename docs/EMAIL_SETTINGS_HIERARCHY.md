# هيكل إعدادات البريد الإلكتروني
## Email Settings Hierarchy

## 📊 المستويات (Hierarchy)

### 1️⃣ إعدادات المنصة (Platform Settings) - **تطبق على الجميع**
**الموقع**: `platform_settings` في Supabase  
**من يتحكم بها**: الأدمن فقط  
**التأثير**: **جميع المستخدمين في المنصة**

```sql
-- هذه الإعدادات تطبق على جميع المستخدمين
SELECT * FROM platform_settings 
WHERE setting_key IN (
  'email_notifications',      -- التحكم الرئيسي (Master Switch)
  'email_new_submission',      -- إشعار عند طلب جديد
  'email_status_change'        -- إشعار عند تغيير الحالة
);
```

**مثال**:
- إذا عطل الأدمن `email_notifications` → **لن يتم إرسال أي بريد لأي مستخدم**
- إذا عطل الأدمن `email_status_change` → **لن يتم إرسال إشعارات تغيير الحالة لأي مستخدم**

---

### 2️⃣ تفضيلات المستخدم الفردية (User Preferences) - **خاصة بكل مستخدم**
**الموقع**: `notification_preferences` في Supabase  
**من يتحكم بها**: كل مستخدم لنفسه  
**التأثير**: **المستخدم الفردي فقط**

```sql
-- هذه الإعدادات خاصة بكل مستخدم
SELECT * FROM notification_preferences 
WHERE user_id = 'user-uuid';
```

**مثال**:
- المستخدم يمكنه تعطيل الإشعارات لنفسه فقط
- لكن إذا كانت إعدادات المنصة معطلة، لن يتم إرسال البريد حتى لو كان المستخدم مفعل إشعاراته

---

## 🔄 منطق التحقق (Check Logic)

### في Edge Functions:

```typescript
// 1. أولاً: التحقق من إعدادات المنصة (Platform Settings)
const emailSettings = await getEmailSettings(supabase);

if (!shouldSendEmail(emailSettings, 'status_change')) {
  // ❌ إعدادات المنصة معطلة → لا ترسل لأي مستخدم
  return;
}

// 2. ثانياً: التحقق من تفضيلات المستخدم (User Preferences)
const userPreferences = await getUserPreferences(userId);

if (!userPreferences.email_enabled) {
  // ❌ المستخدم عطل إشعاراته → لا ترسل له
  return;
}

// ✅ كل شيء مفعل → يمكن الإرسال
await sendEmail(...);
```

## 📋 جدول المقارنة

| الميزة | إعدادات المنصة | تفضيلات المستخدم |
|--------|----------------|-------------------|
| **الموقع** | `platform_settings` | `notification_preferences` |
| **من يتحكم** | الأدمن فقط | كل مستخدم لنفسه |
| **التأثير** | جميع المستخدمين | المستخدم الفردي |
| **الأولوية** | ✅ أولاً (Master Switch) | ثانياً |
| **مثال** | تعطيل جميع الإشعارات | تعطيل إشعاراتي فقط |

## 🎯 أمثلة عملية

### مثال 1: الأدمن يعطل إشعارات تغيير الحالة
```
الأدمن: email_status_change = false
المستخدم: status_change_email = true (مفعل)

النتيجة: ❌ لن يتم إرسال البريد
السبب: إعدادات المنصة لها الأولوية
```

### مثال 2: الأدمن يفعل الإشعارات، المستخدم يعطلها
```
الأدمن: email_notifications = true
المستخدم: email_enabled = false (معطل)

النتيجة: ❌ لن يتم إرسال البريد
السبب: المستخدم عطل إشعاراته
```

### مثال 3: كلاهما مفعل
```
الأدمن: email_status_change = true
المستخدم: status_change_email = true

النتيجة: ✅ سيتم إرسال البريد
```

---

## 🔐 الأمان

- **إعدادات المنصة**: فقط الأدمن يمكنه تعديلها
- **تفضيلات المستخدم**: كل مستخدم يعدل لنفسه فقط
- **RLS Policies**: محمية بـ Row Level Security

---

## 💡 التوصيات

1. **استخدم إعدادات المنصة** للتحكم العام في النظام
2. **اسمح للمستخدمين** بالتحكم في إشعاراتهم الفردية
3. **احترم الأولوية**: إعدادات المنصة أولاً، ثم تفضيلات المستخدم

---

## 📝 ملاحظات

- إعدادات المنصة هي **Master Switch** - إذا كانت معطلة، لن يتم إرسال أي بريد
- تفضيلات المستخدم تعمل فقط إذا كانت إعدادات المنصة مفعلة
- هذا التصميم يوفر مرونة كاملة للأدمن في التحكم في النظام
