# Epic 6: Welcome Emails System
## نظام إرسال الإيميلات الترحيبية

---

## Overview
نظام شامل لإرسال إيميلات ترحيبية للمستخدمين الجدد في المنصة. يتضمن معالجة على دفعات، آلية إعادة المحاولة، وتسجيل شامل لجميع الإيميلات.

## Business Value
- تحسين تجربة المستخدم الجديد من خلال الترحيب الفوري
- توفير معلومات واضحة عن ميزات المنصة
- تتبع شامل لجميع الإيميلات المرسلة
- ضمان موثوقية الإرسال من خلال آلية إعادة المحاولة

---

## Technical Implementation

### الملف الرئيسي
**الموقع:** `supabase/functions/send-welcome-emails/index.ts`

### الوظائف الرئيسية

#### 1. جلب المستخدمين
- جلب جميع المستخدمين من جدول `users`
- تصفية المستخدمين (إزالة التكرارات والإيميلات غير الصالحة)
- حد أقصى: 10,000 مستخدم (safety cap)

#### 2. تصفية المستخدمين
**الخطوات:**
- تنظيف الإيميلات (trim وإزالة المسافات)
- تصفية الإيميلات غير الصالحة (يجب أن تحتوي على @)
- تحويل الإيميلات إلى lowercase لتوحيدها
- استخدام Map لإزالة التكرارات (نأخذ آخر مستخدم لكل إيميل)

#### 3. تسجيل الإيميلات
- تسجيل جميع الإيميلات في `email_log` بحالة `queued`
- تحديث الحالة إلى `sent` أو `failed` حسب نتيجة الإرسال
- حفظ رسائل الخطأ في حالة الفشل

#### 4. معالجة على دفعات (Batch Processing)
**الاستراتيجية:**
- تقسيم المستخدمين إلى دفعات بحجم `BATCH_SIZE` (افتراضي: 2)
- إرسال الإيميلات داخل كل دفعة بشكل متسلسل
- تأخير بين كل إيميل (`DELAY_BETWEEN_EMAILS`: 600ms)
- تأخير بين الدفعات (`DELAY_BETWEEN_BATCHES`: 1000ms)

**مثال:**
- إذا كان `BATCH_SIZE = 2` و `DELAY_BETWEEN_EMAILS = 600ms`
- فإن معدل الإرسال سيكون حوالي 2 إيميل/ثانية

#### 5. آلية إعادة المحاولة (Retry Mechanism)
**الاستراتيجية:**
- عدد المحاولات: `MAX_RETRY + 1` (3 محاولات إجمالاً)
- Exponential Backoff: التأخير يزداد بشكل أسي بين المحاولات
- الحد الأقصى للتأخير: 10 ثواني

**جدول التأخيرات:**
- المحاولة 1: فورية (0ms)
- المحاولة 2: بعد 2 ثانية (2000ms)
- المحاولة 3: بعد 4 ثواني (4000ms)

**الصيغة:**
```
delay = min(2000 * 2^(attempt-1), 10000)
```

#### 6. إرسال الإيميلات
- استخدام Edge Function `send-notification-email` لإرسال الإيميلات
- يدعم عدة خدمات إيميل: Supabase Email, Resend, SendGrid
- التحقق من نجاح الإرسال من خلال استجابة الـ function

## Environment Variables

### متغيرات مطلوبة
- `SUPABASE_URL`: رابط مشروع Supabase
- `SUPABASE_SERVICE_ROLE_KEY`: مفتاح الخدمة للوصول الكامل

### متغيرات اختيارية (قيم افتراضية)
- `FROM_EMAIL`: البريد الإلكتروني المرسل (افتراضي: `rafrs2030@gmail.com`)
- `FROM_NAME`: اسم المرسل (افتراضي: `منصة نشر الأبحاث العربية`)
- `BATCH_SIZE`: حجم الدفعة - عدد الإيميلات المرسلة معاً (افتراضي: `2`)
- `DELAY_BETWEEN_EMAILS`: التأخير بين الإيميلات بالميلي ثانية (افتراضي: `600`)
  - يسمح بحوالي 2 إيميل/ثانية
- `DELAY_BETWEEN_BATCHES`: التأخير بين الدفعات بالميلي ثانية (افتراضي: `1000`)
  - يعطي وقت إضافي بين الدفعات لتجنب تجاوز حدود المعدل

## Functions Documentation

### 1. `generateWelcomeEmailHtml(usernameOrEmail: string): string`
**الوظيفة:**
- توليد إيميل ترحيبي باللغة العربية مع دعم RTL
- تصميم متجاوب (responsive) يعمل على جميع الأجهزة
- استخدام ألوان العلامة التجارية للمنصة (#3D5A94)
- تضمين رابط للدخول إلى المنصة

**المعاملات:**
- `usernameOrEmail`: اسم المستخدم أو البريد الإلكتروني (يستخدم في التحية)

**العائد:**
- `string`: محتوى HTML كامل للإيميل

**التصميم:**
- Header: خلفية متدرجة باللون الأزرق (#3D5A94)
- Content: نص ترحيبي مع قائمة بالميزات
- Button: زر للدخول إلى المنصة
- Footer: حقوق النشر مع السنة الحالية

---

### 2. `logEmailToDatabase(...): Promise<void>`
**الوظيفة:**
- تسجيل كل محاولة إرسال إيميل في قاعدة البيانات
- تتبع حالة الإيميل: `queued`, `sent`, `failed`
- حفظ رسائل الخطأ في حالة الفشل
- تسجيل وقت الإرسال عند النجاح

**المعاملات:**
- `userId`: معرف المستخدم (يمكن أن يكون null)
- `recipientEmail`: البريد الإلكتروني المستقبل
- `subject`: موضوع الإيميل
- `status`: حالة الإيميل (`"sent" | "failed" | "queued"`)
- `errorMessage`: رسالة الخطأ (اختياري، فقط في حالة failed)

**ملاحظات:**
- لا يرمي الخطأ إذا فشل التسجيل (يستخدم `console.warn` فقط)
- هذا يمنع توقف عملية الإرسال بسبب مشاكل في التسجيل
- `sent_at` يتم تعيينه فقط عند `status === "sent"`

---

### 3. `sendEmail(...): Promise<any>`
**الوظيفة:**
- تستدعي Edge Function `send-notification-email` لإرسال الإيميل
- تدعم عدة خدمات إيميل: Supabase Email, Resend, SendGrid
- تتحقق من نجاح الإرسال من خلال استجابة الـ function

**المعاملات:**
- `to`: البريد الإلكتروني المستقبل
- `subject`: موضوع الإيميل
- `html`: محتوى الإيميل بصيغة HTML
- `userId`: معرف المستخدم (اختياري) لتسجيل الإيميل في email_log

**العائد:**
- `Promise<{success: boolean, data?: any}>`: نتيجة الإرسال

**يُرمي:**
- `Error`: في حالة فشل الإرسال

**ملاحظات:**
- الـ function يتوقع استجابة بصيغة `{ success: true/false, error?: string }`
- إذا كان `success === false`، يتم رمي خطأ
- إذا لم يكن هناك data، نعتبره نجاح (بعض Edge Functions لا ترجع data)

---

### 4. `Deno.serve(async (req: Request) => {...})`
**الوظيفة الرئيسية:**
1. جلب جميع المستخدمين من جدول `users`
2. تصفية المستخدمين (إزالة التكرارات والإيميلات غير الصالحة)
3. تسجيل جميع الإيميلات في `email_log` بحالة `queued`
4. إرسال الإيميلات على دفعات (batches) مع احترام حدود المعدل (rate limits)
5. إعادة المحاولة عند الفشل (retry mechanism) مع exponential backoff
6. تحديث حالة الإيميلات في `email_log` (`sent`/`failed`)

**الميزات:**
- معالجة على دفعات لتجنب تجاوز حدود المعدل
- تأخير بين الإيميلات وبين الدفعات
- آلية إعادة المحاولة مع exponential backoff
- تسجيل شامل لجميع الإيميلات في قاعدة البيانات
- استخدام Edge Function `send-notification-email` لإرسال الإيميلات

**CORS:**
- يدعم `OPTIONS` requests
- يسمح بـ `POST` فقط للطلبات الفعلية
- Headers: `Access-Control-Allow-Origin: *`

**Response Format:**
```json
{
  "ok": true,
  "total": 100,
  "sent": 95,
  "failed": 5,
  "results": [
    { "email": "user@example.com", "status": "sent" },
    { "email": "user2@example.com", "status": "failed", "detail": "Error message" }
  ]
}
```

## Error Handling

### أنواع الأخطاء المعالجة

1. **أخطاء قاعدة البيانات:**
   - فشل جلب المستخدمين
   - فشل تسجيل الإيميل في `email_log`
   - لا توقف العملية، يتم تسجيل التحذير فقط

2. **أخطاء إرسال الإيميل:**
   - فشل استدعاء `send-notification-email`
   - فشل الإرسال من خدمة الإيميل
   - يتم إعادة المحاولة تلقائياً (3 محاولات)

3. **أخطاء عامة:**
   - أخطاء غير متوقعة
   - يتم إرجاع رسالة خطأ واضحة في الاستجابة

## Rate Limiting Strategy

### الحدود المطبقة
- **معدل الإرسال:** حوالي 2 إيميل/ثانية
- **حجم الدفعة:** 2 إيميل (قابل للتعديل عبر `BATCH_SIZE`)
- **التأخير بين الإيميلات:** 600ms (قابل للتعديل عبر `DELAY_BETWEEN_EMAILS`)
- **التأخير بين الدفعات:** 1000ms (قابل للتعديل عبر `DELAY_BETWEEN_BATCHES`)

### الحساب
```
معدل الإرسال = 1000 / DELAY_BETWEEN_EMAILS
مثال: 1000 / 600 = 1.67 إيميل/ثانية ≈ 2 إيميل/ثانية
```

## Database Schema

### جدول `email_log`
يستخدم لتسجيل جميع الإيميلات المرسلة:

```sql
CREATE TABLE email_log (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id UUID REFERENCES users(id),
  email_type TEXT NOT NULL, -- 'system', 'notification', etc.
  recipient_email TEXT NOT NULL,
  subject TEXT NOT NULL,
  status TEXT NOT NULL, -- 'queued', 'sent', 'failed'
  error_message TEXT,
  sent_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

**الحالات:**
- `queued`: الإيميل في قائمة الانتظار
- `sent`: تم إرسال الإيميل بنجاح
- `failed`: فشل إرسال الإيميل
